前端代码指南

记录一些前端代码的最佳实践。

HTML

语义化

HTML5提供给我们大量的语义化标签用来精确的描述我们要展示的内容，请使用它丰富的语义化标签来让我们受益。

确保你理解了你要使用的语义化标签，用错误的方式使用了语义化标签比不使用该标签更加糟糕。

简洁化

让你的代码保持简洁，忘记你以前老式的XHTML编码习惯

易读性

易读性不是编码完成后再来想起做，你没有必要成为一个WCGA的专家来改善你的网站，你可以马上做一些很小的事情来获得巨大的改善，比如说：

  • 学习正确的使用alt属性
  • 确保你的a标签和button标签都不是像这样的（<div class=button>）
  • 不是仅仅通过颜色来传递信息，可以加适当动词
  • 明确标记表单控件

语言

虽然定义语言和编码格式是可选得，但是建议在文档声明时同时声明它们，即使它们也会在你的HTTP头部指定，最好使用UTF-8编码而不是用其他字符编码。

性能

除非你有足够的理由要在页面加载前加载你的script脚本，否则不要阻止你的页面的渲染。如果你的样式表非常大，将它隔离出来放在一个单独的样式表中，通过另一个link标签二次加载。尽量减少HTTP请求数量，两个HTTP请求速度明显比一个HTTP请求慢。

CSS

分号

尽管分号是CSS的分割符，但是请始终将它看作一个终止符。

盒子模型

盒子模型理论上应对于整个文档都相同，最好是有一个全局的设置*｛box-sizing: border-box;｝。尽量不要改变特定元素的默认盒子模型。

流

尽量不要改变元素的行为，尽可能让元素处于正常的文档流中。比如说，去除图片周围的文字不应该让你改变它的默认显示样式。

同样，尽量不要让一个元素脱离正常的文档流

定位

有许多的方法定位CSS元素，但是尽量限制为下面的属性/值。按优先顺序排列：

选择器

减少选择器的耦合,考虑添加一个类名不要超过3个伪类。

你不需要的选择器避免重载

权重

不要让值和选择器难以重写，尽量少使用id和!important

覆写

覆写样式使选择器和调试更加困难，尽可能的避免覆写。

继承

不要重复声明可以继承的样式

简洁

使你的代码保持简洁，使用尽量少的属性，不是非常需要避免使用多个属性。

语义化

尽可能使用英语而不是数字

浏览器前缀

去除过时的浏览器前缀，如果确实需要使用浏览器前缀，在标准写法前面写入浏览器前缀

动画

最好使用transitions而不是animations，除了opacity和transform避免使用动画的其它属性。

单位

尽可能使用无单位的值，如果需要使用相对单位最好使用rem。最好使用秒而不是毫秒做单位。

颜色

如果你需要透明度，请使用rgba。否则的话，始终使用16进制格式。

绘制

如果资源很容易绘制，尽量少使用HTTP请求图片资源。

Hacks

不要使用Hacks。比如对于浏览器硬件加速的触发。

Javascript

性能

好的可读性，准确性和表现性的重要程度超过性能，javascript基本上不会存在性能瓶颈。性能优化一般是图像压缩，网络接入和DOM重排这也方面。选择你记住的任何一个准则来优化性能。

无状态性

尽量让你的方法保持纯净，所有的方法在理想情况下应该都没有副作用，使用内部数据并且返回一个新的对象而不是使用外部的数据。

原生方法

尽量多的使用原生方法

强制转换

当隐式的强制转换有意义时，使用隐式的强制转换。否则，不要使用隐式的强制转换

循环

不要使用循环因为他们强制你使用可变对象，请使用Array.prototype方法。

如果不能这样使用，或者不适合使用Array.prototype方法，可以使用递归

这里有一个通用递归方法使递归更加容易使用

参数

忘记argument对象，...<params>始终是更好的选择，因为：
1.它是有命名的，所以它让你更好的了解函数需要的参数。
2.它是一个真实的数组，这使得它更容易被使用

Apply

忘记Apply()方法，使用spread operator替代

Bind

不要使用bind()方法绑定作用域，因为有更好的做法。

嵌套函数

当你不用嵌套函数时，避免嵌套函数

Composition

Avoid multiple nested function calls. Use composition instead.

缓存

在大量的数据和比较昂贵的操作，可以使用缓存功能。

变量

尽量使用const而不是let，尽量使用let而不是var

条件判断

最好使用IIFE函数来包裹if语句来返回结果，不仅是if语句，switch语句也可以。

对象迭代

尽量避免使用for...in进行对象迭代

使用Map取代Object

尽管有时使用object很合适，但是maps通常是一个更好的，更强大的选择，当你对使用object还是map有疑问时，请使用’Map’。

柯里化

柯里化非常强大，是许多国外开发人员的典范，但是请不要在不合适的时候滥用柯里化。

可读性

不要使用看似聪明的技巧混淆你代码的意图。

代码重用

不要害怕创建一些小的，高度可组合，可重用的方法。

依赖性

减少依赖关系，第三方的代码是你不知道的，不要因为几个容易复制的代码就加载整个代码库。

总结

这次系统的了解了前端代码的一些最佳实践，尤其是包含了许多es2015的最佳写法，翻译完这篇博客后给我的感觉的对前端的代码编写认知又更近了一步。
